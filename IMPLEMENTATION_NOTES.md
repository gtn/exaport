# Exaport Backup/Restore and Navigation Implementation

## Overview

This PR implements two major improvements to the Exaport block plugin:

1. **Navigation and Page Layout Fix** for `category_distribution.php`
2. **Backup/Restore Support** for category and view distribution templates

## Problem Statement

### 1. Navigation Issue
The `category_distribution.php` page was accessible from the block but had several issues:
- Missing from the main Exaport navigation tabs
- Used incorrect page layout (`incourse` instead of standard)
- Did not follow the same page setup pattern as other pages like `importexport.php`
- Inconsistent header and navigation compared to other Exaport pages

### 2. Missing Backup/Restore
The plugin had no backup/restore implementation, meaning:
- When a course was backed up and restored, distribution templates were lost
- Teachers/admins could not copy "template courses" with pre-configured category/view structures
- Auto-distribution settings were not preserved during course copy operations

## Solution Implemented

### 1. Navigation and Page Layout Fix

**Changes to `lib/lib.php`:**
- Added `category_distribution` tab to the main navigation
- Implemented capability check (`block/exaport:distributecategories`) before displaying the tab
- Ensures only users with proper permissions see the tab

**Changes to `category_distribution.php`:**
- Removed manual `$PAGE->set_pagelayout('incourse')` configuration
- Removed manual `$PAGE->set_title()` and `$PAGE->set_heading()` calls
- Now uses `block_exaport_print_header("category_distribution")` for consistent navigation
- Added proper wrapper div handling with `block_exaport_wrapperdivend()`
- Removed duplicate heading that was previously rendered

**Result:**
- "Preset structure" tab now appears in navigation for authorized users
- Page layout is consistent with `importexport.php` and other Exaport pages
- Full navigation breadcrumbs and tabs are properly displayed

### 2. Backup/Restore Implementation

Created standard Moodle 5.0 backup/restore structure for block plugins:

**Backup Files Created:**

1. **`backup/moodle2/backup_exaport_block_task.class.php`**
   - Main backup task class extending `backup_block_task`
   - Defines backup steps and file areas
   - No file areas needed (pure relational data)

2. **`backup/moodle2/backup_exaport_stepslib.php`**
   - Defines backup structure using `backup_nested_element`
   - Specifies data sources for three tables:
     - `block_exaport_course_templ` - Category distribution templates
     - `block_exaport_view_templ` - View distribution templates
     - `block_exaport_templ_dist` - Distribution settings
   - Scoped to course context using `backup::VAR_COURSEID`

**Restore Files Created:**

3. **`backup/moodle2/restore_exaport_block_task.class.php`**
   - Main restore task class extending `restore_block_task`
   - Defines restore steps
   - No file areas to restore

4. **`backup/moodle2/restore_exaport_stepslib.php`**
   - Implements restore structure step
   - Processes each table type with proper ID remapping:
     - **Category templates**: Maps parent IDs (`pid`) to maintain hierarchy
     - **View templates**: Simple course ID update
     - **Distribution settings**: Handles unique constraint (one per course)
   - Updates all `courseid` fields to target course
   - Preserves all attributes (names, descriptions, flags, sortorder)

## Technical Details

### Tables Backed Up

1. **`block_exaport_course_templ`** (Category Distribution Templates)
   - Hierarchical structure with parent-child relationships
   - Fields: id, courseid, pid (parent id), name, sortorder, share_to_teachers, timemodified
   - Requires special handling for `pid` field during restore

2. **`block_exaport_view_templ`** (View Distribution Templates)
   - Flat list structure
   - Fields: id, courseid, name, description, sortorder, share_to_teachers, timemodified

3. **`block_exaport_templ_dist`** (Distribution Settings)
   - One record per course (unique constraint on courseid)
   - Fields: id, courseid, auto_distribute, auto_distribute_views, timemodified
   - Controls auto-distribution behavior on student enrolment

### ID Remapping Strategy

**During Restore:**
- All `courseid` fields are updated to the target course ID
- Category parent IDs (`pid`) are remapped:
  - Mapping table tracks old ID → new ID
  - Parent categories are processed before children (maintains hierarchy)
  - If parent not found, defaults to root (pid = 0)
- New record IDs are generated by database auto-increment

### Backup Process Flow
1. Moodle backup system discovers `backup_exaport_block_task.class.php`
2. Task executes `backup_exaport_block_structure_step`
3. Step queries database for records matching courseid
4. XML structure is generated with nested elements
5. Data is written to backup archive as `exaport.xml`

### Restore Process Flow
1. Moodle restore system discovers `restore_exaport_block_task.class.php`
2. Task reads `exaport.xml` from backup archive
3. For each record type:
   - `process_course_template()` - Creates categories with ID mapping
   - `process_view_template()` - Creates views
   - `process_distribution_setting()` - Creates/updates settings
4. ID mappings allow proper parent-child relationships
5. Data is written to target course database tables

## Known Limitations

1. **No file attachments**: These tables contain only relational data, no files
2. **Course-scoped only**: Does not backup user portfolios or items
3. **Minimal backup**: Only backs up distribution templates, not actual distributed items
4. **Auto-distribution trigger**: Requires student enrolment to trigger after restore
